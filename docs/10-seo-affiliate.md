# 10. SEO設計・アフィリエイト導線・計測設計

## 1. SEO戦略

### 1.1 基本方針(2026年の検索環境を前提に)

- **E-E-A-T特化**: AI量産コンテンツが減点される環境で、「実購入・実写真・使用期間・気になった点」の一次体験データが構造的に全ページへ入る設計(07章スキーマで強制)そのものがSEO戦略。
- **AI Overviews / LLM検索対応**: 結論を先に書く記事構造+FAQ・構造化データで、AI要約に引用されやすい形を標準化(引用元リンク=新しい流入)。
- **ロングテール狙い**: ドメインパワーがない初期は「商品名+レビュー」「シーン+おすすめ」(例: 沖縄 子連れ 持ち物)のロングテールに集中。ビッグワードは追わない。
- **鮮度シグナル**: `updatedAt` を運用(年次の「買ってよかった」更新、使用期間バッジの更新=再クロール促進)。

### 1.2 キーワード設計(コンテンツタイプ別)

| タイプ | 狙うクエリ型 | 例 |
|---|---|---|
| product / article:review | 「{商品名} レビュー/口コミ/デメリット」 | 「◯◯ 水筒 漏れる」 |
| article:roundup | 「{シーン} おすすめ/持ち物/必需品」 | 「子連れ沖縄 持ち物」 |
| article:comparison | 「{A} {B} 違い/どっち」 | — |
| article:ranking | 「買ってよかった {年}/{カテゴリ}」 | 「買ってよかった育児グッズ 2026」 |
| article:log | 指名・回遊用(SEO非依存) | — |

### 1.3 テクニカルSEO実装仕様

| 項目 | 実装 |
|---|---|
| メタ | title 32字/description 120字(スキーマで上限強制)。テンプレート: 商品=`{name}を{usagePeriod}使ったレビュー | {siteName}` |
| 構造化データ | 商品: `Product`+`Review`(author=個人名, reviewRating)/ 記事: `Article` / 全下層: `BreadcrumbList` / roundup: `ItemList` / About: `Person`+`Organization`。実装は `lib/seo.ts` に集約 |
| canonical | 全ページ自己参照。パラメータ付きURLは正規URLへ |
| sitemap / RSS | `@astrojs/sitemap` 自動生成 / `rss.xml`(記事) |
| OGP | ビルド時に satori で自動生成(写真+タイトル+ロゴ)。SNSシェアの見た目=ブランド |
| 内部リンク | 逆引きインデックス(07章)による自動相互リンク+AI内部リンク提案(09章 Stage5) |
| 画像SEO | alt必須(Zod)・ファイル名は内容ベース・AVIF/WebP・遅延読み込み(Hero除く) |
| robots | draft/review系はそもそも出力されない(noindexに頼らない) |

## 2. アフィリエイト導線設計

### 2.1 導線の全体マップ

```
[流入] SNS(ストーリー/リール/Threads) ─┐
       検索(レビュー/まとめクエリ)  ─┼→ [記事] ─ ProductEmbed ─→ [商品ページ] ─→ Yahoo!
       AI検索の引用                    ─┘     └──────(直接)──────────────────────↗
```

- 収益ページは実質「商品ページ」1種類に集約。記事は商品ページへの送客と信頼形成に徹する(ProductEmbed内ボタンは購入意欲が固まった読者用のショートカット)。
- Yahoo!トラベル: 旅行記(article:log/roundup)内に宿の ProductEmbed(mall=yahoo-travel)。「泊まった宿」も商品エンティティとして登録する(category=travel)。

### 2.2 コンプライアンス実装(必須)

| 要件 | 実装 |
|---|---|
| ステマ規制(景表法) | アフィリエイトリンク含有ページ冒頭+ボタン直下に PrLabel 自動表示(コンポーネント強制・人力不要) |
| ポリシー明文化 | `/disclosure/`: 掲載基準(実際に使ったものだけ)・報酬の仕組み・価格注記 |
| リンク属性 | `rel="nofollow sponsored noopener"` 統一(AffiliateButtonで強制) |
| Yahoo!規約 | リンクは管理画面発行のものを無改変で使用(人間が貼る運用: 09章 Stage6)。禁止表現(最安値保証等)はvoice.mdの禁止語に追加 |
| 価格表記 | 「購入時価格」と明記し現在価格はリンク先確認を促す(二重価格・誤認防止) |

### 2.3 CTA最適化ルール(V1固定・V3で計測後に見直し)

- 商品ページ: ヘッダー内1+文末1(5.4章)。色は墨色固定、サイズlg、full width(モバイル)。
- ボタン上に「参考価格 ¥3,280(2026年6月購入時)」を表示: 価格が分かるとCTRが上がり、リンク先での離脱が減る。
- 「気になった点」セクションの直後に文末CTAを置く(正直な情報の直後が最も信頼が高い位置)。

## 3. 計測設計

### 3.1 V1(公開初日から・実装コストほぼゼロ)

| 指標 | 取得元 |
|---|---|
| PV・流入元(SNS/検索別) | Cloudflare Web Analytics(UTM付与は5.5章) |
| 検索クエリ・掲載順位 | Google Search Console + Bing Webmaster |
| クリック→成果・報酬 | Yahoo!アフィリエイト管理画面 |
| Core Web Vitals | CrUX / Lighthouse CI(ビルド毎) |

### 3.2 V3(自前計測: Cloudflare Workers + D1)

- `/go/{productId}/{mall}?pos={position}` リダイレクタWorker: クリックをD1に記録→302で本来のアフィリエイトURLへ。
- 記録: productId / mall / pos / referer(ページ) / UA種別 / 日時。個人情報は持たない(IPは保存しない)。
- 週次ActionsがD1を集計し `content-hub/analytics/weekly.json` を生成 → AIが「今週のレポート+再投稿候補+テコ入れ提案」をIssueで報告。
- ダッシュボード: 専用UIは作らない。週次Issueレポートで十分(作るならV4でPagesに管理ページ)。

### 3.3 意思決定に使うKPIツリー

```
月間報酬
├─ 商品ページ到達数
│   ├─ SNS流入(ストーリーリンクタップ率 × 投稿頻度)
│   └─ 検索流入(インデックス数 × 平均順位)
├─ アフィリエイトCTR(目標8%)
│   └─ ページ品質(読了率・「気になった点」の説得力)
└─ リンク先CVR(Yahoo!側・間接影響のみ)
    └─ 送客の質(購入意欲の高い文脈で送っているか)
```

改善の優先順は常に「インデックス数(コンテンツ量)→ SNS導線 → CTR微調整」。V1〜V2はコンテンツ量産(=AIパイプライン稼働率)がすべて。
