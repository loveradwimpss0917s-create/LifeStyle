# 00. プロジェクト概要・現状分析・競合分析・戦略提言

## 1. プロジェクト定義

| 項目 | 内容 |
|---|---|
| プロジェクト名 | LIFESTACK |
| 種別 | ライフスタイルブランド(サイト + SNS + AIコンテンツパイプライン) |
| 収益モデル | Yahoo!ショッピング公式アフィリエイト / Yahoo!トラベル(将来: Amazon・楽天) |
| オーナーの投入リソース | 「普段通り生活する + ストーリーを投稿する」のみ |
| AIの役割 | ストーリー等の日常発信を、レビュー・記事・SNS投稿・SEO資産へ変換する |
| 成功の定義 | (1) オーナーの作業がストーリー投稿+承認のみで回る (2) コンテンツが資産として蓄積し検索流入が伸び続ける (3) 世界観が一貫したブランドとして認知される |

## 2. 現状分析

### 2.1 保有アセット

| チャネル | フォロワー | 特性 |
|---|---|---|
| Instagram(写真) | 約1,700 | 最大アセット。世界観の核。ビジュアル訴求力あり |
| Instagram(家) | 約330 | 暮らし・家電・日用品と親和性が高い |
| YouTube | 約30 | ほぼ未開拓。Shorts転用先として温存 |
| TikTok | 約10 | ほぼ未開拓。台本転用先として温存 |
| Webサイト | 0 | 本プロジェクトで新規構築 |

### 2.2 オーナーの行動特性(設計上の最重要制約)

- **できること**: ストーリー投稿(ほぼ毎日)、写真撮影、日常の消費行動(旅行・育児用品・家電・コーヒー)
- **できないこと(継続しないこと)**: フィード投稿、リール制作、記事執筆
- **帰結**: システムは「ストーリー投稿を唯一の入力」として成立しなければならない。フィード・リール・記事はすべて出力(AI生成+人間承認)である。

### 2.3 SWOT

| | プラス | マイナス |
|---|---|---|
| **内部** | S: 写真力・毎日のストーリー習慣・実生活に根ざした商品体験(旅行/育児/家電) | W: 長文コンテンツ制作が続かない・動画編集リソースなし・ドメインパワーゼロ |
| **外部** | O: Yahoo!ショッピングアフィリエイトは競合が楽天/Amazonより少ない・AI生成コストの急低下・「顔の見える個人の実体験レビュー」への需要増(Google E-E-A-T) | T: AI量産型アフィサイトへの検索エンジンの締め付け・SNSアルゴリズム変動・アフィリエイト規約変更 |

**戦略的含意**: AI量産サイトが淘汰される時代において、勝ち筋は「実体験 × 実写真 × 一貫した世界観」。AIは執筆代行ではなく、**本人の実体験を構造化して多チャネルに展開する編集者**として位置づける。これはGoogleのE-E-A-T(経験・専門性・権威性・信頼性)評価とも合致する。

## 3. 競合分析

### 3.1 直接競合(個人ライフスタイル系アフィリエイトメディア)

| 類型 | 代表例 | 強み | 弱み(=LIFESTACKの機会) |
|---|---|---|---|
| ブログ型レビューサイト | WordPress個人ブログ多数 | SEO記事量 | デザインが「ブログ然」としていて世界観がない。広告過多で信頼感が低い |
| 楽天ROOM系キュレーター | 楽天ROOM上位ユーザー | 手軽さ・モール内流入 | プラットフォーム依存。自前ブランド資産にならない |
| Instagramアフィリエイター | 暮らし系インスタグラマー | フォロワーとの関係性 | ストック資産(検索流入)がない。投稿が流れて消える |
| 大手比較メディア | my-best、360LiFE | 網羅性・検証力 | 個人の顔・生活感がない。「この人の暮らしが好き」という愛着が生まれない |

### 3.2 世界観リファレンス(競合ではなくベンチマーク)

| ブランド | 学ぶべき点 | LIFESTACKへの translate |
|---|---|---|
| Apple | 余白と階層。1画面1メッセージ。写真=ヒーロー | セクションごとに1つの主役写真。テキストは短く |
| 無印良品 | 「感じ良い暮らし」の思想。商品ではなく生活を売る | カテゴリを商品分類ではなく生活文脈(旅・育児・朝のコーヒー)で編む |
| Notion | 静かなUI。モノクロ基調+機能美 | UIクロームを消し、コンテンツだけを見せる |
| Aesop | 高級感のあるタイポグラフィ・落ち着いた色 | セリフ体の英字見出し+抑制されたアクセントカラー |
| Kinfolk / Cereal Magazine | Webマガジンとしての誌面設計 | 記事を「特集」として編む。グリッドと大判写真 |

### 3.3 ポジショニング

```
        世界観・ブランド重視
              ▲
              │  ● LIFESTACK(目標)
   Kinfolk ●  │
              │      ● 暮らし系インスタグラマー
──────────────┼──────────────▶ 収益化・商品情報重視
              │
  個人ブログ ●│  ● my-best / 比較メディア
              │
        情報羅列・広告的
```

**ポジション定義**: 「Webマガジンの世界観 × 個人の実体験レビュー × AI駆動の多チャネル展開」。この3つを同時に満たすプレイヤーは(2026年時点で)ほぼ存在しない。

## 4. ブランド戦略

### 4.1 ブランドステートメント(案)

> 日常を、美しく残す。
> LIFESTACK は、旅と子育てと暮らしの記録から生まれた、小さなライフスタイルマガジンです。

- サイト名は仮に **LIFESTACK** とする(実装時に変数1箇所で変更可能にする。→ 12章 実装仕様)
- タグライン: 「日常を、美しく残す。」(英: *Stacking life, beautifully.*)

### 4.2 コンテンツの3層構造

| 層 | 内容 | 役割 | 生成主体 |
|---|---|---|---|
| Layer 1: Life(生活) | ストーリー投稿・写真・購入行動 | 素材。真正性の源泉 | 人間(オーナー) |
| Layer 2: Asset(資産) | 商品データ・レビュー・記事・使用ログ | ストック。検索流入とブランドの中身 | AI生成 + 人間承認 |
| Layer 3: Reach(拡散) | リール・Threads・TikTok・Shorts・再投稿 | フロー。新規接点の獲得 | AI生成 + 人間承認 |

すべてのコンテンツはLayer 2(サイト上の資産)を正本(Single Source of Truth)とし、SNSはそこからの派生とする。**SNSが主・サイトが従ではなく、サイトが正本・SNSが配信網**。これが「ブランドサイト」たる所以である。

## 5. 要望に対する積極的提案(採用/変更/削除の提言)

ユーザー要望をそのまま実装せず、長期運用性の観点から以下を提言する。**すべて本設計書には提言反映済み**。

### 5.1 変更提案

| # | 元の要望 | 提案 | 理由 |
|---|---|---|---|
| P-1 | ストーリー投稿をAIが自動解析して商品DB登録 | **V1では「iPhoneから写真+ひとことメモを所定フォームで送る」半自動に変更**。完全自動解析はV3 | Instagram Graph APIのストーリー取得は権限審査が重く、個人開発の初期段階では投資対効果が低い。「撮った写真を選んで3行メモを書く」は1分で終わり、AI解析の精度も桁違いに上がる |
| P-2 | 一商品から10種のコンテンツを一括生成 | **生成は一括、公開は人間の承認制(Draft→Review→Publish)** | AI全自動公開はブランド毀損リスク(誤情報・規約違反・ステマ規制)が大きい。承認は「読んでOKを押す」だけの設計にし、負荷は最小化する |
| P-3 | お気に入り機能(訪問者向け) | **V1はlocalStorageベースの軽量実装、アカウント機能は作らない** | 静的サイトの利点(速度・保守ゼロ)を守る。ログイン機能はコスト対効果が最悪 |
| P-4 | 検索機能 | **Pagefind(静的検索)を採用。サーバー不要** | 静的サイトのまま全文検索が可能。Algolia等の外部SaaSは月額コストと依存が発生する |
| P-5 | PWA対応 | **V1ではmanifest+アイコンのみ(インストール可能)。オフラインキャッシュはV3** | Service Workerのキャッシュ戦略は更新事故の温床。まずは価値が薄い |
| P-6 | 分析(クリック数等) | **V1はCloudflare Web Analytics + アフィリエイトリンクへのUTM/計測パラメータ付与のみ。自前計測基盤はV3(Workers)** | 初期は計測より生産。Yahoo側の管理画面レポートで十分に意思決定できる |

### 5.2 削除(スコープ外)提案

| # | 要望 | 判断 | 理由 |
|---|---|---|---|
| P-7 | AIチャット(訪問者向け) | V4以降に保留 | 個人メディア規模では利用率が低くコストだけ発生。ニュースレターの方がLTVに効く |
| P-8 | 価格比較(モール横断) | V3以降に保留 | 各モールAPIの規約・在庫価格の鮮度管理が重い。静的サイトと相性が悪い |

### 5.3 追加提案(要望になかったが必要なもの)

| # | 提案 | 理由 |
|---|---|---|
| P-9 | **「記録(Log)」コンテンツタイプの新設** — 商品に紐づかない日常の記録(旅行記・週報)も資産化する | 「商品ページ+レビュー記事」だけのサイトはアフィサイトに見える。商品が絡まない読み物が世界観と回遊とE-E-A-Tを作る |
| P-10 | **ステマ規制(景表法)対応をテンプレートに組み込む** — 全ページにPR表記コンポーネントを標準搭載 | 2023年10月施行の日本のステマ規制により、アフィリエイトリンクを含むページへの「PR」明示は法的義務。ブランド信頼にも直結 |
| P-11 | **コンテンツハブ(content hub)アーキテクチャ** — 商品1件を中心に、全派生コンテンツ(SNS投稿・記事)をIDで相互リンクするデータ構造 | 「一度作ったコンテンツを何度でも使う」思想の実装形。再投稿候補の自動抽出もこの構造の上に成立する |
| P-12 | **ニュースレターはV2で導入(buttondown等)** | SNSアルゴリズム・検索変動に依存しない自前リストは、5年スパンでは最重要資産 |

## 6. 収益設計の基本方針

1. **収益はブランドの副産物**。ページ内のアフィリエイトCTAは1商品ページにつき主ボタン1つ+文末1つまで。バナー広告・ポップアップは永久に禁止。
2. **Yahoo!ショッピング中心の理由を活かす**: 5のつく日・買う買うサンデー等のセール文脈はSNS(ストーリー・Threads)で消化し、サイトは非時限の資産コンテンツに徹する。
3. **リンク管理は一元化**(affiliate link resolver): 商品データにはモール別URLを保持し、表示時にコンポーネントが解決する。将来Amazon/楽天追加時も商品データに1フィールド足すだけ。
4. **計測可能性**: すべてのアフィリエイトリンクに `?ref=lifestack&pos={page}_{position}` 形式の内部識別子を付与(Yahoo側パラメータ仕様に準拠)し、どのページ・どの位置が成果を生むかをV3で分析可能にする。
